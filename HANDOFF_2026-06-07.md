## What Was Fixed

### 1. OAuth redirect mismatch
- Root cause: Supabase/ops config was sending Google OAuth back to an old deployment URL.
- Confirmed by user landing on:
  - `https://coastercount-pro-my5my4q2dq-uw.a.run.app/`
- Repo config was updated and Cloud Run was redeployed.
- User later confirmed Google login worked again in browser/Safari.

### 2. Blank screen / startup crash
- Earlier crash path was around startup and provider initialisation.
- Fixed:
  - top-level error boundary placement
  - removed old `importmap` React collision risk from `index.html`
  - added missing `useMemo` import in `context/AppContext.tsx`
  - hardened storage fallbacks in `services/storage.ts`
  - added boot/runtime diagnostics in `index.tsx` and `context/appLifecycle.ts`

### 3. Mobile Safari post-login crash
- User reported iPhone Safari crashed after login with:
  - `Script error.`
  - `Boot stage: init:complete`
- This showed auth/data init was finishing and the crash was in the logged-in render path.
- Implemented iPhone Safari compatibility fallbacks:
  - simplified dashboard analytics on iPhone Safari
  - simplified yearly growth view on iPhone Safari
  - removed some heavy shell effects for iPhone Safari
  - simplified Profile heatmap on iPhone Safari
  - blocked heavier secondary views in iPhone Safari compatibility mode:
    - `PARK_STATS`
    - `RANKINGS`
    - `MILESTONES`
    - `GAME`
    - `QUEUE_HUB`
- Result: normal Safari usage became stable.

### 4. Add to Home Screen crash
- User reported Safari crashed when using `Share -> Add to Home Screen`.
- Cause was likely Safari snapshotting the full heavy page.
- Added lightweight install mode via:
  - `?install=1`
- Important regression happened once:
  - I temporarily moved install-mode logic into `index.html` in a way that broke the main Vite entry, producing a blank blue screen on the normal app URL.
  - This was fixed by restoring the normal module entry in `index.html` and moving install-mode rendering into `index.tsx`.

### 5. Home Screen icon opening 403
- User could add to Home Screen, but tapping the icon gave `403 forbidden`.
- This was fixed in two stages:
  1. `index.tsx` install mode now redirects to `/` when launched in standalone mode.
  2. `manifest.json` was corrected to:
     - `"start_url": "/"`
     - `"scope": "/"`
     - `"display": "standalone"`
- User had to delete and recreate the Home Screen icon after this.
- Final user confirmation: **â€śworks nowâ€ť**.

## Current User Workflow

This is the workflow the user asked to clarify right before handoff:

- Use `Safari` to sign in with Google.
- After Safari login succeeds, the `Home Screen` app can be used normally.
- Safari does **not** need to remain open.
- If the user signs out or the session expires, they must go back to `Safari` to log in again.
- The Home Screen app intentionally does **not** allow Google login directly.

This is by design now because standalone iOS/PWA Google OAuth was unstable and/or vulnerable to `403 disallowed_useragent`.
